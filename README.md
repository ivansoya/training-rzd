# YOLO Dataset Manager

Веб-приложение для загрузки, валидации и просмотра статистики датасетов в формате YOLO.

- **Бэкенд:** Python + Flask
- **Фронтенд:** TypeScript + React (Vite)

## Возможности

- Загрузка датасета в виде `.zip` архива (кнопка в скрываемом боковом меню слева).
- Валидация архива на соответствие формату YOLO:
  - наличие `*.yaml` конфигурации с `names` / `nc`;
  - наличие папок `images` и `labels`;
  - проверка каждого файла разметки `.txt` (5 значений в строке, нормализованные координаты `[0,1]`, корректный id класса).
- Распаковка валидного архива в отдельную папку.
- Датасеты разделены на **«Загруженные»** и **«Аугментированные»**, которые бэкенд
  хранит в разных папках:
  - `backend/data/uploaded/<name>` — загруженные пользователем;
  - `backend/data/augmented/<name>` — создаются **только самим приложением**.
- Боковое меню сворачивается; удаление датасета — кнопкой `✕` напротив каждого.
- Статистика выбранного датасета справа.
- **Аугментации (albumentations):**
  - раздел «Настроить аугментации» с редактором конфигураций;
  - конфигурация — это набор аугментаций с полным набором параметров каждой
    (параметры извлекаются интроспекцией из установленной версии albumentations);
  - есть встроенная базовая конфигурация без аугментаций (изменить/удалить нельзя);
  - окно превью: выбранная конфигурация применяется к случайному изображению
    датасета (с отрисовкой bbox, которые корректно трансформируются вместе с
    картинкой);
  - конфигурации хранятся на сервере (`backend/data/configs.json`).
- **Несколько конфигураций за раз:** при создании аугментации можно выбрать
  сразу несколько конфигураций — каждая выполняется как отдельный полный проход
  по изображениям датасета (N конфигураций → N проходов). Файлы из разных
  проходов получают суффикс (`x.aug1.png`, `x.aug2.png`, …) и складываются в один
  датасет.
- **Выбор охвата:** аугментировать **все изображения** или **только train**.
- **Прогресс-бары:** при загрузке датасета показывается прогресс передачи архива
  (после 100% — статус «Передача архива на сервер…», т.к. данные ещё идут по
  сети) и прогресс распаковки; при генерации аугментаций — сколько изображений
  обработано. Длинные операции выполняются на сервере как фоновые задачи, а
  интерфейс опрашивает их статус (`GET /api/jobs/<id>`).
- **Статистика считается один раз** после распаковки (показывается отдельным
  статусом «Подсчёт статистики…» без прогресс-бара) и кэшируется в
  `<датасет>/.stats.json`. Поскольку датасеты в программе неизменяемы,
  последующие открытия статистики мгновенны.
- **Обучение YOLO-моделей** (раздел «🧠 Обучение моделей»):
  - запуск обучения на выбранном датасете (загруженном или аугментированном);
  - выбор версии модели из доступных на сервере; свои модели (`.pt`/`.yaml`)
    можно добавлять и удалять;
  - настройка гиперпараметров (epochs, imgsz, batch, lr0, optimizer, patience и
    др.) и **выбор устройства — CPU или GPU** (если запущен GPU-образ);
  - **все аугментации YOLO принудительно отключены** при обучении;
  - **реальное время через SSE:** сервер сам пушит обновления клиенту
    (Server-Sent Events, `GET /api/trainings/<id>/stream`) — фронтенд ничего не
    опрашивает. Обновляются статус, текущая эпоха, метрики по эпохам (mAP50,
    mAP50-95, precision, recall, losses), а также **прогресс по итерациям внутри
    эпохи** (текущий батч / всего, скорость it/s и текущие box/cls/dfl losses),
    как в консольной полосе YOLO. После батчей эпохи показывается **прогресс
    валидации** (`Валидация N/M`) — видно, чего приложение «ждёт» в паузе после
    эпохи. Прогресс собирается через штатные коллбэки ultralytics
    (`on_train_epoch_start` / `on_train_batch_end` / `on_val_batch_end`);
  - **история всех обучений** со статистикой; обучение можно остановить;
  - **веса** (`best.pt`) сохраняются на сервере, их можно скачать и удалить
    (или удалить обучение целиком). Всё хранится в `backend/data/trainings/` и
    переживает перезапуск;
  - каждое обучение выполняется в отдельном процессе, поэтому «Остановить» и
    «Удалить» немедленно завершают его (вместе с дочерними процессами).
- **Проверка моделей (инференс)** (раздел «Проверка моделей»):
  - **библиотека видео с каталогами (проектами):** видео загружаются в
    именованный каталог (`backend/data/videos/`, метаданные в `meta.json`) и
    группируются по нему в боковом списке. Загружать и **просматривать видео в
    плеере можно независимо от того, есть ли обученные модели** (плеер
    поддерживает перемотку через HTTP range);
  - загрузка видео сопровождается прогресс-баром передачи;
  - когда модель есть — выбираешь её и запускаешь инференс по уже загруженному
    видео: сервер прогоняет кадры, рисует детекции и кодирует результат в H.264
    (через ffmpeg). Результат проигрывается на странице **с пометкой, какой
    моделью он обработан**;
  - статистика: число кадров, детекций, кадров с детекциями, детекций на кадр и
    разбивка по классам со средней уверенностью;
  - у каждого видео — своя история проверок (каждая помечена моделью); видео и
    его проверки удаляются (удаление видео каскадно убирает его проверки).
- Приложение запоминает, **из какого датасета** и **какими конфигурациями**
  создана каждая аугментация. Сохраняются **сами значения** конфигураций
  (снимок параметров), поэтому запись не теряется, даже если исходные
  конфигурации потом удалить. Хранится также пользовательское отображаемое имя
  (`backend/data/augmented.json`). Все такие датасеты лежат в отдельной папке
  `backend/data/augmented/` и доступны для просмотра наравне с загруженными.

## Ожидаемая структура архива

```
my-dataset.zip
├── data.yaml
├── images/
│   ├── train/...
│   └── val/...
└── labels/
    ├── train/...
    └── val/...
```

`data.yaml`:

```yaml
train: images/train
val: images/val
names:
  0: big-grab
  1: hook
  ...
nc: 20
```

Файл разметки `labels/train/img1.txt` (нормализованные координаты):

```
12 0.625721 0.691786 0.086057 0.235000
4 0.656250 0.246786 0.423958 0.220714
```

## Запуск через Docker (рекомендуется)

```bash
docker compose up --build
```

Откройте **http://localhost:8080**.

- `frontend` — nginx, отдаёт собранный React и проксирует `/api` на бэкенд.
- `backend` — Flask + gunicorn.

### Где хранятся данные

Все датасеты, конфигурации, обучения и метаданные хранятся в **именованном
Docker-томе** `yolo-data` (смонтирован в `/app/data`). Том лежит в файловой
системе ext4 внутри ВМ Docker, а **не** через bind mount на диск Windows. Это
важно для производительности: на Docker Desktop/Windows bind mount идёт через
медленный слой обмена файлами, из-за чего загрузчик данных при обучении не
успевает кормить GPU (видеокарта простаивает в ожидании картинок). Именованный
том даёт почти нативную скорость I/O.

Данные **переживают пересборку образов** (`docker compose up --build`) и
`docker compose down`. Стираются они только при `docker compose down -v`.

```bash
# посмотреть том
docker volume ls | grep yolo-data
# сделать бэкап в tar
docker run --rm -v training-rzd_yolo-data:/data -v "$PWD":/out alpine \
  tar czf /out/yolo-data-backup.tar.gz -C /data .
```

Остановить: `docker compose down` (данные остаются).

### Обучение на GPU

По умолчанию образ собирается с CPU-версией PyTorch. Для обучения на GPU есть
override-файл `docker-compose.gpu.yml`, который собирает бэкенд с CUDA-сборкой
torch и пробрасывает в контейнер все GPU:

```bash
docker compose -f docker-compose.yml -f docker-compose.gpu.yml up --build
```

Требования на хосте:
- NVIDIA GPU и свежий драйвер;
- [NVIDIA Container Toolkit](https://docs.nvidia.com/datacenter/cloud-native/container-toolkit/latest/install-guide.html)
  (чтобы Docker мог отдать GPU контейнеру).

После запуска на странице «Обучение моделей» в поле **«Устройство»** появятся
доступные GPU; обучение с выбранным GPU идёт на сервере (в контейнере). Если
GPU-образ не запущен, доступен только CPU, а запрос обучения на GPU отклоняется.

> Образ с CUDA-сборкой torch заметно больше (несколько ГБ) и собирается дольше.

> **Shared memory.** Обучение использует несколько процессов загрузки данных
> (DataLoader), которые обмениваются батчами через `/dev/shm`. По умолчанию в
> Docker он всего 64 МБ, из-за чего возникает ошибка
> `bus error ... insufficient shared memory (shm)`. В `docker-compose.yml` для
> бэкенда задан `shm_size: "2gb"` — этого достаточно для типовых конфигураций.
> Для очень больших `batch`/`imgsz` увеличьте значение. Изменение `shm_size`
> применяется при `docker compose up` (пересборка образа не нужна — контейнер
> просто пересоздаётся).

## Запуск без Docker (для разработки)

### 1. Бэкенд

```bash
cd backend
python -m venv .venv
# Windows
.venv\Scripts\activate
# Linux/macOS
source .venv/bin/activate

pip install -r requirements.txt
python app.py
```

Сервер поднимется на `http://127.0.0.1:5000`.

### 2. Фронтенд

```bash
cd frontend
npm install
npm run dev
```

Откройте `http://localhost:5173`. Запросы `/api/*` проксируются на Flask (см. `vite.config.ts`).

## API

| Метод  | Путь                          | Описание                                         |
| ------ | ----------------------------- | ------------------------------------------------ |
| GET    | `/api/datasets`               | список датасетов (с полем `kind`)                |
| POST   | `/api/datasets`               | загрузка `.zip` (multipart `file`) → возвращает `{job_id}` |
| GET    | `/api/datasets/<kind>/<name>` | статистика датасета                              |
| DELETE | `/api/datasets/<kind>/<name>` | удаление датасета                                |
| POST   | `/api/datasets/augmented`     | создать аугментацию `{source, config_ids, display_name, scope}` → `{job_id}` |
| GET    | `/api/jobs/<id>`              | статус фоновой задачи (прогресс/результат)       |
| GET/POST/DELETE | `/api/models[/<id>]` | версии моделей (список / добавить `.pt`/`.yaml` / удалить) |
| GET/POST | `/api/trainings`            | список обучений / запустить обучение → `{id}`    |
| GET    | `/api/trainings/<id>`         | состояние обучения (метрики по эпохам)            |
| GET    | `/api/trainings/<id>/stream`  | SSE-поток: сервер пушит состояние (эпохи + итерации) |
| POST   | `/api/trainings/<id>/stop`    | остановить обучение                              |
| DELETE | `/api/trainings/<id>`         | удалить обучение вместе с весами                 |
| GET/DELETE | `/api/trainings/<id>/weights` | скачать / удалить веса `best.pt`            |
| GET    | `/api/videos`                 | список видео в библиотеке (с каталогом)           |
| GET    | `/api/videos/catalogs`        | список каталогов (проектов)                       |
| POST   | `/api/videos`                 | загрузить видео (multipart `file` + `catalog`) → запись |
| GET    | `/api/videos/<id>/file`       | отдать видео (range/перемотка) — для плеера       |
| DELETE | `/api/videos/<id>`            | удалить видео (каскадно удаляет его проверки)     |
| GET    | `/api/inference/models`       | обученные модели, доступные для инференса         |
| GET    | `/api/inference[?video_id=]`  | список проверок (опц. фильтр по видео)            |
| POST   | `/api/inference`              | запустить инференс `{video_id, model_run_id}` → `{id}` |
| GET/DELETE | `/api/inference/<id>`     | состояние проверки / удалить                      |
| GET    | `/api/inference/<id>/video`   | размеченное видео (H.264, поддержка перемотки)    |
| GET    | `/api/aug/transforms`         | реестр доступных аугментаций и их параметров     |
| GET    | `/api/aug/configs`            | список конфигураций                              |
| POST   | `/api/aug/configs`            | создать конфигурацию                            |
| PUT    | `/api/aug/configs/<id>`       | изменить конфигурацию                           |
| DELETE | `/api/aug/configs/<id>`       | удалить конфигурацию                            |
| POST   | `/api/aug/preview`            | превью `{source, transforms}` → base64 PNG       |

`kind` — это `uploaded` или `augmented`.
